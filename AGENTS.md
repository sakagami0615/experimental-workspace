# AGENTS.md

このリポジトリは「実験環境」を複数置き、それらをモノレポのように扱うことを目的とします。何を実験するかは流動的なので、構成は抽象化して汎用的に保ちます。個々の実験は独立して開発・検証でき、共通のガードレール（コーディング規約・セキュリティ・CI）で品質を担保します。

## 推奨ディレクトリ構成

- `experiments/`：実験ごとのルートフォルダを格納。
	- `experiments/<name>/`：各実験は必ず1ディレクトリにまとめる。
		- `src/`（必要の場合のみ）：ソースコード（言語ごとにサブ構成を設けてよい）。
		- `tests/`（必要の場合のみ）：ユニット/統合テスト。
		- `docs/`（必要の場合のみ）：実験の目的、使い方、再現手順。
		- `data/`（必要の場合のみ）：小さなサンプルのみ。大容量データは外部ストレージを使う（詳細は下へ）。
		- `scripts/`（必要の場合のみ）：起動・実験実行スクリプト。
		- `.env.example`（必要の場合のみ）：環境変数のテンプレ。実際の `.env` はコミットしない。
        - `.gitignore`：実験ごとにignoreしたい内容を記載する。
        - `AGENTS.md`：実験ごとに設定したいルール等を記載する。
		- `README.md`：実験の概要と最小再現手順。

- `tools/`（任意）：共通ツール・スクリプト。
- `common/`（任意）：実験間で再利用するコードやユーティリティ。

## 設計方針

- 各実験は可能な限り独立させる（依存は明示的に）。
- 再現性を重視する：環境、依存関係、ラン時設定を文書化する。
- 小さく始める：実験は最小の動作単位で作り、拡張で拡張する。

## コーディング規約とドキュメンテーション

- 総則：すべてのコードはリント・フォーマッタを通すこと。言語別の推奨は `pyproject.toml` / `package.json` 等で指定する。
- Python固有：PEP8（フォーマッタは `black` 推奨）と型アノテーション (`mypy` 等) を推奨。
- ドックストリング／ドキュメント：すべての公開関数・クラス・モジュールにドックストリングを必須とする。

	- Python のドックストリング（推奨フォーマット、例）

		"""
		簡単な説明（1行）。

		詳細な説明（必要なら複数行）。

		Args:
				param1 (int): 説明。
				param2 (str, optional): 説明。

		Returns:
				bool: 成功/失敗。

		Raises:
				ValueError: 条件を満たさない場合。
		"""

- 他言語についてはその言語の標準的なドキュメンテーションスタイル（Javadoc/TSDoc等）に従うこと。

## Git・セキュリティのガードレール

- 自動で Git 操作を行わないこと：エージェントや自動化スクリプトは、明示的な承認がない限り `git commit` / `git push` 等を実行してはなりません。
- 絶対にコミットしてはいけないもの（例）: 実運用のシークレット、`*.env`、`*.pem`、`*.key`、`credentials`、大容量バイナリ/データ（`data/raw/` など）。
- `.env.example` を用意し、実際の `.env` はローカルのみで管理する。
- リポジトリルートに次のような `.gitignore` を用意すること（例）:

	- `.env`
	- `*.key`
	- `*.pem`
	- `data/`
	- `*.sqlite3`
	- `__pycache__/`

- 秘密情報検出ツールを CI と pre-commit に組み込む（例：`detect-secrets`, `pre-commit` の `truffleHog` 等）。
- 大きなファイルの誤コミット対策：GitHub の LFS、DVC、または外部ストレージを利用し、CI で意図しない大ファイルをブロックする。

## データの扱い

- 生データ（大容量）はリポジトリに置かない。参照用のメタデータや小さなサンプルのみコミットする。
- 実験でデータが必要な場合はダウンロードスクリプトを用意し、`README.md` に取得手順を記載する。

## CI／ローカルチェック（推奨）

- `pre-commit` を導入して以下をチェックすることを推奨：フォーマット、静的解析、型チェック、シークレット検出、禁止ファイルの存在チェック。
- CI では PR ごとに次を必須にする：テスト通過、リンター無エラー、シークレット検出パス、タイプチェック（言語に応じて）。

## PR/レビューのルール

- **説明**: 何をしたか・なぜかを `PR` 説明に明記。
- **再現手順**: 必要なセットアップ手順やコマンドを `README.md` に追記。
- **テスト**: 新規機能はテストを追加し、既存テストはグリーンになること。
- **ドキュメント**: 影響範囲のドキュメント（docstring や `docs/`）を更新。
- **セキュリティ**: シークレット・大容量ファイルが含まれていないことを確認。

## 運用上の注意

- 自動化スクリプトやエージェントは、リポジトリへ変更を与える前に必ずヒトの確認を要求すること。
- 自動でシークレットや大容量データをアップロードしないこと。自動化は検出と警告に留めるべき。
